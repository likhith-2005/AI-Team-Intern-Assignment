# B3 – Honest Goodput Analysis

## Misreading in REPORT_v0

The report incorrectly interprets the **reported_tok_s** column as the actual throughput delivered to users.

However, **reported_tok_s** is the benchmark harness's throughput counter. It counts all generated tokens but does not directly represent the user-visible serving rate when requests experience different waiting times and scheduler behavior.

This misinterpretation leads to two incorrect conclusions:

1. Longer prompts provide better throughput.
2. Batch size 48 would achieve approximately 3200 tok/s.

Both conclusions are based on reading the `reported_tok_s` column without checking whether the values are consistent with the measured execution time.

---

## Honest goodput (Method 1)

For the batch-24 long-prompt run:

- Number of requests = 24
- Generated tokens per request = 512
- Wall-clock time = 61.16 s

Total generated tokens

= 24 × 512

= 12,288 tokens

Goodput

= 12,288 ÷ 61.16

≈ 200.9 tokens/s

---

## Honest goodput (Method 2)

The benchmark also reports:

- End-to-end latency (p95) ≈ 69.22 s

Using the observed completion time,

Goodput

≈ 12,288 ÷ 69.22

≈ 177.5 tokens/s

This provides an independent estimate of the delivered throughput experienced by users. The value is lower because it includes the effect of request completion latency.

---

## Correct conclusion

The benchmark does **not** demonstrate that longer prompts improve serving throughput.

Instead, increasing the prompt length significantly increases KV-cache usage and memory pressure. Throughput initially improves with batching but begins to decrease once scheduler preemption occurs.

Therefore, the report should conclude that batch size 24 is the best operating point for this workload. Increasing the batch size beyond this point increases KV-cache contention and scheduler preemption, reducing effective user-visible throughput instead of improving it.
