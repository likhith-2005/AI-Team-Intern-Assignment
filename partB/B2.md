# B2 – Throughput Anomaly Analysis

## Observed anomaly

For the long-context benchmark (prompt length = 3584), throughput initially increases with batch size but then decreases despite larger batches.

| Batch | Throughput (tok/s) | KV Utilization | Preempted Sequences |
|------:|-------------------:|---------------:|--------------------:|
| 4 | 565.4 | 0.16 | 0 |
| 8 | 902.6 | 0.31 | 0 |
| 16 | 1311.4 | 0.62 | 0 |
| 24 | 1607.4 | 0.93 | 0 |
| 32 | 1384.0 | 0.97 | 7 |
| 48 | 1298.5 | 0.97 | 23 |

The anomaly is that throughput peaks at **batch size 24** and then drops at batch sizes **32** and **48**, even though more requests are being processed.

---

## Explanation

At batch size 24, KV-cache utilization reaches approximately **93%**, and no sequences are preempted.

At batch sizes 32 and 48, KV-cache utilization increases to approximately **97%**, causing the scheduler to preempt active sequences.

These preemptions introduce additional scheduling overhead and reduce effective throughput. The GPU is no longer compute-limited; instead, it becomes constrained by KV-cache capacity.

Therefore, increasing the batch size beyond 24 does not improve throughput because memory pressure outweighs the benefits of batching.

---

## Proposed deployment change

Configure the serving system to limit the maximum batch size to approximately **24** for this workload.

### Predicted effect

Current throughput:

- Batch 24 → **1607.4 tok/s**
- Batch 32 → **1384.0 tok/s**
- Batch 48 → **1298.5 tok/s**

Using batch size 24 avoids scheduler preemption while delivering the highest measured throughput.

Compared with batch 48, throughput improves by approximately:

\[
\frac{1607.4 - 1298.5}{1298.5} \times 100 \approx 23.8\%
\]

while also reducing request latency and eliminating scheduler preemption.
