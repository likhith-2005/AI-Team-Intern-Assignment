# B2 – Throughput Anomaly Analysis

## Observed anomaly

For the long-context workload (prompt length = 3584 tokens), throughput initially increases as the batch size increases, but then decreases even though more requests are processed simultaneously.

| Batch Size | Throughput (tok/s) | KV Cache Utilization | Preempted Sequences |
|-----------:|-------------------:|---------------------:|--------------------:|
| 4 | 565.4 | 0.16 | 0 |
| 8 | 902.6 | 0.31 | 0 |
| 16 | 1311.4 | 0.62 | 0 |
| 24 | 1607.4 | 0.93 | 0 |
| 32 | 1384.0 | 0.97 | 7 |
| 48 | 1298.5 | 0.97 | 23 |

The throughput reaches its maximum at **batch size 24** and then decreases at **batch sizes 32 and 48**, which is the observed anomaly.

---

## Explanation

At batch size **24**, KV-cache utilization reaches approximately **93%**, and no sequences are preempted.

At batch sizes **32** and **48**, KV-cache utilization increases to approximately **97%**, causing the scheduler to preempt active sequences.

These preemptions introduce additional scheduling overhead and increase memory pressure. As a result, the GPU is no longer primarily limited by compute resources; instead, performance becomes constrained by KV-cache capacity.

Consequently, increasing the batch size beyond **24** does not improve throughput because the overhead introduced by KV-cache contention outweighs the benefits of processing additional requests concurrently.

---

## Proposed deployment change

Reduce the effective maximum concurrent sequences (for example, by lowering the scheduler's maximum batch size or `max_num_seqs`) so that KV-cache utilization remains below approximately **93%** for this workload.

---

## Predicted effect

Current benchmark results:

- Batch 24 → **1607.4 tok/s**
- Batch 32 → **1384.0 tok/s**
- Batch 48 → **1298.5 tok/s**

Keeping KV-cache utilization below the point where scheduler preemption begins is expected to:

- Eliminate scheduler preemption (from **23** preempted sequences to **0**).
- Increase throughput from **1298.5 tok/s** to approximately **1607.4 tok/s**, representing an improvement of approximately:

\[
\frac{1607.4 - 1298.5}{1298.5} \times 100 \approx 23.8\%
\]

- Improve median time-to-first-token (TTFT) because requests no longer wait for preempted sequences.
- Maintain higher overall serving efficiency by avoiding excessive KV-cache contention.
