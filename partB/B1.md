# B1 – KV Cache Capacity Calculation

## (a) KV-cache bytes per token

From the model specification:

- Layers = 28
- KV heads = 8
- Head dimension = 128
- KV precision = FP16 (2 bytes)
- Each token stores both **Key** and **Value**

Formula:

Bytes per token

= 2 × Layers × KV Heads × Head Dimension × FP16 bytes

= 2 × 28 × 8 × 128 × 2

= 114,688 bytes

≈ 112 KB per token

---

## (b) Maximum concurrent 4096-token sequences

GPU memory = 24 GB

Available GPU memory

= 24 × 0.92

= 22.08 GB

Runtime overhead

= 1.6 GB

Memory available for KV cache

= 22.08 − 1.6

= 20.48 GB

Memory required per sequence

= 4096 × 114,688

= 469,762,048 bytes

≈ 448 MB

Maximum concurrent sequences

= 20.48 GB ÷ 448 MB

≈ 45 sequences

---

## (c) Comparison with benchmark log

Observed benchmark:

| Batch | KV Utilization | Preempted Sequences |
|------:|---------------:|--------------------:|
| 24 | 0.93 | 0 |
| 32 | 0.97 | 7 |
| 48 | 0.97 | 23 |

The theoretical calculation predicts that the GPU can support approximately 45 concurrent 4096-token sequences before exhausting KV-cache memory.

The benchmark is consistent with this prediction. At batch size 24, there is no scheduler preemption. At batch sizes 32 and 48, KV-cache utilization reaches approximately 97%, preemption begins, and throughput decreases despite larger batches, indicating that the serving system is operating near its KV-cache capacity.
